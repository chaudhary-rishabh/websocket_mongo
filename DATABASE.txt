1. CONNECTION SETUP
────────────────────────────────────────────────────────────────────────────────

File: src/config/db.ts

  import mongoose from 'mongoose'

  export async function connectDB() {
    await mongoose.connect(env.MONGODB_URI, {
      // Mongoose 9 uses the new Node driver defaults; no deprecated options needed
    })
    logger.info('MongoDB connected')
  }

  - Single connection per process lifecycle (Node.js is single-threaded)
  - Mongoose manages a connection pool internally (default: 5 connections)
  - On bootstrap failure: process exits; graceful shutdown closes the connection

Configuration:
  MONGODB_URI = mongodb://localhost:27017/chatapp  (development)
  — In production: use a connection string with auth, replica set, and TLS

All schemas use { timestamps: true } which automatically adds:
  createdAt: Date   — set on document creation
  updatedAt: Date   — updated on every save (except AiMessage which uses timestamps: { createdAt: true, updatedAt: false })

Document IDs:
  All _id fields are MongoDB ObjectId (Mongoose default)
  Interface declarations use mongoose.Types.ObjectId for type safety
  When passed over HTTP: serialized as 24-char hex strings
  When used in queries: always cast via new mongoose.Types.ObjectId(stringId)

2. COLLECTION: users
────────────────────────────────────────────────────────────────────────────────

File: src/modules/users/user.model.ts
Mongoose model name: 'User'

SCHEMA
  email:            String, required, unique, lowercase, trim
  username:         String, required, unique, trim, minlength:3, maxlength:30
  displayName:      String, required, trim, maxlength:60
  passwordHash:     String, required, select:false
                    — NEVER returned in queries unless explicitly .select('+passwordHash')
                    — bcrypt hash (12 rounds for registration, never stored as plain text)
  avatar:           String (Cloudinary URL), optional
  bio:              String, maxlength:300, optional
  role:             String, enum:['user','admin'], default:'user'
  isOnline:         Boolean, default:false
                    — Toggled by WebSocket connect/disconnect via setOnlineStatus()
  lastSeen:         Date, default:Date.now
                    — Updated to current time when user goes offline
  refreshTokenHash: String, select:false
                    — bcrypt hash (10 rounds) of the current valid refresh token
                    — Set on login/register/refresh; cleared on logout
                    — Used for refresh token rotation and revocation

INSTANCE METHOD
  comparePassword(candidate: string): Promise<boolean>
    return bcrypt.compare(candidate, this.passwordHash)
    — Called in auth.service login to verify credentials

TOSON TRANSFORM
  toJSON: { transform: (doc, ret) => { delete ret.passwordHash; delete ret.refreshTokenHash } }
  — Automatically strips sensitive fields from any user.toJSON() call
  — Means you can safely pass user objects to res.json() without leaking hashes

INDEXES
  { email: 1 }         — unique constraint (enforced at DB level too)
  { username: 1 }      — unique constraint (enforced at DB level too)
  NOTE: unique:true on the schema already creates a unique index in MongoDB.
  The explicit index() calls are for query optimization (non-unique lookup speed).

QUERY PATTERNS
  Login:          User.findOne({ email }).select('+passwordHash')
  Register check: User.findOne({ $or: [{ email }, { username }] }).lean()
  Refresh token:  User.findById(sub).select('+refreshTokenHash')
  Search:         User.find({ _id:{$ne:me}, $or:[{username:/q/i},{displayName:/q/i}] }).limit(20)
  Online status:  User.findByIdAndUpdate(id, { $set: { isOnline, lastSeen? } })

RELATIONSHIPS
  - Referenced by: Conversation.members[], Conversation.admins[], Conversation.createdBy
  - Referenced by: Message.senderId, Message.reactions[].userId, Message.readBy[].userId
  - Referenced by: AiSession.userId, AiMessage.userId

3. COLLECTION: conversations
────────────────────────────────────────────────────────────────────────────────

File: src/modules/conversations/conversation.model.ts
Mongoose model name: 'Conversation'

SCHEMA
  type:       String, enum:['dm','group'], required
              — 'dm' = direct message (exactly 2 members, no name)
              — 'group' = group chat (2+ members, has name)

  name:       String, trim, maxlength:80, optional
              — Only used for group conversations

  avatar:     String (Cloudinary URL), optional
              — Group avatar

  members:    [ObjectId ref:'User'], required
              — Array of all member IDs. This is the primary membership list.
              — For DMs: always length 2
              — For groups: 2 to N

  admins:     [ObjectId ref:'User']
              — Subset of members who have admin rights
              — Creator is always the first admin
              — If last admin leaves, the first remaining member is auto-promoted
              — Admin rights: update group name/avatar, add/remove members

  lastMessage: subdocument {
    content:  String
    senderId: ObjectId ref:'User'
    type:     String enum:['text','image','voice','file']
    timestamp: Date
  }
              — Denormalized snapshot of the most recent message
              — Updated on every sendMessage() in message.service.ts
              — Used by the sidebar to show last message preview without joining messages

  createdBy:  ObjectId ref:'User', required

INDEXES
  { members: 1 }      — primary access pattern: find all convs for a userId
  { updatedAt: -1 }   — secondary: sort by most recently active (sidebar ordering)

WHY MEMBERS AS ARRAY (not a separate join table)
  MongoDB is document-oriented. Embedding members[] as an array of ObjectIds:
  - Enables the $all operator for DM dedup: { members: { $all: [A, B], $size: 2 } }
  - The { members: 1 } index makes { members: userId } queries O(log n) against the index
  - Members are populated via .populate() when needed, or kept as IDs for membership checks

DM DEDUPLICATION
  Before creating a DM, the service checks:
  Conversation.findOne({ type:'dm', members:{ $all:[userId1, userId2], $size:2 } })
  — $all ensures both users are in the array
  — $size:2 ensures no group convs with those same 2 users match
  — If found, returns the existing conv (idempotent create)

QUERY PATTERNS
  List user convs:    Conversation.find({ members: userId }).sort({ updatedAt:-1 })
  Get single conv:    Conversation.findById(id).populate('members','username displayName avatar isOnline lastSeen')
  Add member:         conv.members.push(newObjectId); conv.save()
  Remove member:      conv.members = conv.members.filter(id => id.toString() !== target); conv.save()
  Update lastMsg:     Conversation.findByIdAndUpdate(id, { $set: { lastMessage:{...}, updatedAt:new Date() } })

RELATIONSHIPS
  - members[]: many-to-many with User (one user in many convs, one conv has many users)
  - admins[]: subset of members[], same relationship
  - lastMessage.senderId: ref to User (denormalized, not joined)
  - Referenced by: Message.conversationId (one-to-many with messages)

4. COLLECTION: messages
────────────────────────────────────────────────────────────────────────────────

File: src/modules/messages/message.model.ts
Mongoose model name: 'Message'

SCHEMA
  conversationId: ObjectId ref:'Conversation', required, index:true
                  — Belongs to exactly one conversation

  senderId:       ObjectId ref:'User', required
                  — Who sent the message. Populated on reads.

  type:           String, enum:['text','image','voice','file'], required
                  — 'text'  = plain text content
                  — 'image' = content is caption, mediaUrl is image URL
                  — 'voice' = content is duration label, mediaUrl is audio URL
                  — 'file'  = content is filename, mediaUrl is download URL

  content:        String, required, maxlength:10000
                  — For text: the message body
                  — For media: descriptive label or empty string

  mediaUrl:       String, optional
                  — Cloudinary URL for image/voice/file messages

  mediaMeta:      subdocument {
    originalName, mimeType, size, duration, width, height
  }               — Optional media metadata for rendering (e.g. image dimensions,
                  audio duration, file size display)

  reactions:      [{ userId: ObjectId ref:'User', emoji: String maxlength:8 }]
                  — Embedded array, toggled via addReaction()
                  — Each user-emoji pair is unique (addReaction removes if already exists)

  readBy:         [{ userId: ObjectId ref:'User', readAt: Date }]
                  — Tracks per-user read status
                  — Populated by markRead() via $addToSet
                  — Used to render read receipts in the UI

  deliveredTo:    [{ userId: ObjectId ref:'User', deliveredAt: Date }]
                  — Delivery receipts (infrastructure ready, not yet wired to UI)

  replyTo:        ObjectId ref:'Message', optional
                  — Points to the parent message for threaded replies

  deletedFor:     [ObjectId ref:'User']
                  — Soft-delete list. If your userId is in this array, you don't see the msg.
                  — Sender delete → hard delete (findByIdAndDelete)
                  — Non-sender delete → push userId to deletedFor ($addToSet)

  editedAt:       Date, optional
                  — Set when a text message is edited; null otherwise

INDEXES
  { conversationId: 1, createdAt: -1 }
  — Compound index: the dominant query is "all messages in conversation X, newest first"
  — conversationId alone would work but adding createdAt makes sort free (no extra sort step)
  — Descending createdAt supports cursor pagination (cursor = oldest message _id fetched)

  Note: index: true on the conversationId field also creates a single-field index.
  The compound index is more specific and covers the single-field queries too, so both
  exist — MongoDB will use the compound one for paginated queries.

SOFT DELETE vs HARD DELETE
  Sender deletes:
    Message.findByIdAndDelete(messageId)   — gone for everyone
  Non-sender deletes (hides from that user only):
    Message.findByIdAndUpdate(messageId, { $addToSet: { deletedFor: userId } })
  Query filter applied in listMessages:
    { deletedFor: { $ne: userId } }        — $ne on an array checks "not in array"

CURSOR-BASED PAGINATION
  GET /messages?cursor=<messageId>&limit=30    (default limit: 30, max: 100)

  filter._id = { $lt: new mongoose.Types.ObjectId(cursor) }
  — ObjectId encodes creation time, so $lt gives "older than this message"
  Sort: { _id: -1 } (newest first), then reverse the results before returning
  — Client receives chronological order; fetching more scrolls backward in time
  hasMore = fetched.length === limit   (exact match — no +1 probe needed)
  nextCursor = String(messages[0]._id)  — the oldest message in this page
  Response envelope: { items, hasMore, nextCursor? }

POPULATE PATTERN
  senderId populated with: 'username displayName avatar'
  — Only the fields needed for rendering; never expose email/passwordHash

QUERY PATTERNS
  List messages:   Message.find({ conversationId, deletedFor:{$ne:userId} })
                     .sort({ _id:-1 }).limit(n).populate(...)
  Send message:    Message.create({ conversationId, senderId, type, content })
  Mark read:       Message.updateMany({ conversationId, senderId:{$ne:userId},
                     'readBy.userId':{$ne:userId} },
                     { $addToSet:{ readBy:{ userId, readAt:new Date() } } })
  Reactions:       msg.reactions.push/splice → msg.save()

5. COLLECTION: aisessions
────────────────────────────────────────────────────────────────────────────────

File: src/modules/ai/ai.session.model.ts
Mongoose model name: 'AiSession'

SCHEMA
  userId:         ObjectId ref:'User', required, index:true
                  — One user can have MANY sessions (not unique).
                  — userId is indexed for fast "find all sessions for user" queries.
                  — NOTE: if migrating from the old single-session schema, drop the
                    stale unique index: db.aisessions.dropIndex("userId_1")

  title:          String, optional, maxlength:80
                  — Auto-set by persist() from the first user message in the session
                    (content.slice(0, 75).trim()). Never overwritten after that.
                  — Used as the session display name in the session list sidebar.

  messageCount:   Number, default:0
                  — Total number of AI turns (user + assistant combined)
                  — Incremented by $inc on every persist() call

  lastActivityAt: Date, default:Date.now
                  — Updated on every message persist; used to sort sessions newest-first

TIMESTAMPS
  { timestamps: true }  — adds createdAt and updatedAt

PURPOSE
  AiSession is the root anchor for a named AI chat thread.
  Each session has its own message history and title.
  Users can create new sessions ("New Chat"), switch between sessions, and delete
  individual sessions or all sessions at once.

QUERY PATTERNS
  List sessions:
    AiSession.find({ userId }).sort({ lastActivityAt: -1 }).lean()

  Create session:
    AiSession.create({ userId, messageCount: 0, lastActivityAt: new Date() })

  Auto-title (first message only):
    AiSession.updateOne(
      { _id: sessionId, title: { $exists: false } },
      { $set: { title: content.slice(0, 75).trim() } }
    )

  Update on message:
    AiSession.findByIdAndUpdate(sessionId, {
      $inc: { messageCount: 1 },
      lastActivityAt: new Date()
    })

  Delete one session:
    AiMessage.deleteMany({ sessionId })
    AiSession.findByIdAndDelete(sessionId)

  Delete all sessions for user:
    sessionIds = (await AiSession.find({ userId })).map(s => s._id)
    AiMessage.deleteMany({ sessionId: { $in: sessionIds } })
    AiSession.deleteMany({ userId })

RELATIONSHIPS
  - userId → User (many-to-one: one user can have N AiSessions)
  - Referenced by: AiMessage.sessionId (one-to-many: session has many messages)

6. COLLECTION: aimessages
────────────────────────────────────────────────────────────────────────────────

File: src/modules/ai/ai.message.model.ts
Mongoose model name: 'AiMessage'

SCHEMA
  sessionId:  ObjectId ref:'AiSession', required, index:true
              — Which AI session this message belongs to

  userId:     ObjectId ref:'User', required, index:true
              — Denormalized from session for direct per-user queries
              — Allows querying "all AI messages for user X" without joining AiSession first

  role:       String, enum:['user','assistant'], required
              — 'user'      = the human's message or analysis request label
              — 'assistant' = the AI's streamed response (fully accumulated)

  content:    String, required, maxlength:50000
              — For role:'user': the message text or analysis request label
              — For role:'assistant': the complete AI response (stored after stream ends)

  type:       String, enum:['chat','analysis'], required, default:'chat'
              — 'chat'     = normal free-form AI conversation turn
              — 'analysis' = a personality/emotion analysis request + response pair

  metadata:   subdocument {
    model:            String  — e.g. 'deepseek-chat'
    analyzedUserId:   String  — ObjectId string of the user who was analyzed
    analyzedUserName: String  — display name of the analyzed user
  }           — Populated only on 'analysis' type messages (assistant turn)
              — Allows the UI to render "AI analyzed Bob" labels without parsing content

TIMESTAMPS
  timestamps: { createdAt: true, updatedAt: false }
  — Messages are immutable once created; no reason to track updatedAt
  — Saves a write operation on every message read

INDEXES
  { sessionId: 1, createdAt: 1 }  — compound index
  — Primary access pattern: all messages for a session in chronological order
  — Used by GET /ai/messages

  Individual single-field indexes: sessionId, userId
  — sessionId index is covered by the compound index (leftmost prefix rule)
  — userId index allows direct queries: AiMessage.find({ userId }) without knowing sessionId

CURSOR PAGINATION (same pattern as messages)
  GET /api/v1/ai/messages?before=<messageId>&limit=100
  filter._id = { $lt: before }   — messages older than cursor
  Sort: { createdAt: 1 }         — chronological order

CONTENT PERSISTENCE STRATEGY
  User message is persisted BEFORE the AI call starts.
  Assistant message is persisted AFTER the stream ends (in the pipeStream onDone callback).
  This means:
  - If the AI call fails → user turn is saved, assistant turn is not → shows partial history
  - If client disconnects mid-stream → server continues accumulating, saves full response
  - If server crashes mid-stream → user message saved, assistant message lost (acceptable)

RELATIONSHIPS
  - sessionId → AiSession (many-to-one: many messages per session)
  - userId → User (many-to-one, denormalized shortcut)
  - Completely isolated from: users.conversations, messages, conversations
    The AI models do NOT reference regular chat collections and vice versa.
    AI routes READ from conversations/messages for analysis input but WRITE only to ai* collections.

9. DESIGN DECISIONS & RATIONALE
────────────────────────────────────────────────────────────────────────────────

DENORMALIZED lastMessage in Conversation
  - Avoids a $lookup or extra query to show the sidebar (most frequent UI operation)
  - Trade-off: the snapshot can be stale if a message is deleted (acceptable; sidebar is a preview)
  - Updated atomically in sendMessage() via findByIdAndUpdate with $set

EMBEDDED reactions[] and readBy[] in Message
  - These are bounded arrays (reactions: one per user-emoji pair, readBy: one per user)
  - In a 1-to-1 or small group, the array stays small
  - Embedding avoids a separate collection and join for every message render
  - For very large groups, this could grow — mitigated by the 10000-char content limit

SELECT:FALSE on sensitive fields
  passwordHash and refreshTokenHash have select:false
  - Mongoose NEVER includes them in query results unless .select('+fieldName') is explicit
  - Prevents accidental exposure in any findOne/find query that isn't specifically auth-related

SOFT DELETE for messages
  - Users expect "delete for me" functionality (WhatsApp pattern)
  - Hard delete would remove the message from the conversation for all participants
  - deletedFor array grows at most to the number of members — bounded and acceptable

ROLLING REFRESH TOKEN
  - On every refresh call, a new refresh token is generated and the old hash is replaced
  - Any previously issued refresh token immediately becomes invalid
  - Logout clears the hash — all refresh tokens revoked at once
  - This prevents refresh token reuse attacks without needing a token blacklist

AI COLLECTION ISOLATION
  - aisessions and aimessages are completely separate from the chat data
  - Purpose: AI conversation history is a different product concern from chat history
  - Isolation makes it easy to: purge AI data independently, scale AI storage separately,
    apply different retention policies, or move AI data to a different database later
  - The AI routes read from conversations/messages for analysis input — this is a
    read-only dependency; no writes cross the boundary

MULTI-SESSION AI ARCHITECTURE
  - userId on AiSession is NOT unique — one user can have many sessions (like chat threads)
  - "New Chat" creates a new AiSession document via POST /ai/sessions
  - Sessions are sorted by lastActivityAt descending; most recent is shown first
  - Session title is auto-populated from the first user message (slice 0-75) so users
    don't have to name sessions manually — the conversation names itself
  - The title write uses { $exists: false } as a guard so subsequent messages in the
    same session never overwrite the title set from the first message
  - Why the change: a single-timeline AI is limiting — users want separate contexts
    for different topics (coding help vs. personal analysis vs. general Q&A) without
    the history of one polluting the other
  - Delete granularity: DELETE /sessions/:id deletes one thread; DELETE /sessions wipes all
  - Old unique index (userId_1 on aisessions) must be dropped on migration:
      db.aisessions.dropIndex("userId_1")
